1. GLFW input handling: 콜백과 폴링

GLFW에서 사용자 인풋을 받아 이벤트를 처리하는 방식을 두 가지로 나누어 놓았다. 공식문서

폴링으로 인풋받기 (Poll driven)

인풋이 들어왔는지 전수조사하는 걸 polling이라 함

learnopengl에서 만든 processInput 함수가 폴링을 사용하는 예이다. 렌더루프 안에

if(glfwGetKey(window, GLFW_KEY_W) == GLFW_PRESS)
	zOffSet += zDelta;

를 넣으면 렌더루프 안의 glfwPollEvents() 가 호출될 때 W키가 눌려 있는 상태인지 폴링한 후, window 에 대해서 GLFW_KEY_W 의 상태를 GLFW_PRESS 로 바꾸라는 이벤트를 실행하기 때문에 if문 안의 코드가 실행될 수 있다고 이해하였다.

콜백으로 인풋 받기 (Event driven)

전역스코프에 다음과 같은 함수를 정의한 뒤,

void key_callback(GLFWwindow* window, int key, int scancode, int action, int mods){
   if(key == GLFW_KEY_ESCAPE && action == GLFW_PRESS)
      glfwSetWindowShouldClose(window, true);
}

렌더 루프 앞에

glfwSetKeyCallback(window, key_callback);

를 넣어주면 된다.

위의 예시의 경우 esc를 누르면 이벤트 큐에 esc를 눌렀다는 이벤트가 등록되고, glfwPollEvents() 가 호출될 때 사전에 set해놓은 콜백함수를 불러 정의부를 실행한다. 아래 후술할 FPS를 0.2정도(5초마다 렌더 한 번)로 두어 확인할 수 있다.

  • 폴링의 경우 렌더루프가 얼마나 빨리 도느냐에 따라 시간당 실행되는 횟수가 다르다. 예를 들어 캐릭터 이동의 경우 프레임 속도를 일정하게 조절해주지 않으면 등속도 운동이 아니게 된다.
  • 매 루프마다 지속적으로 상태를 확인해서 실행시켜야 하는 코드가 있다면 (캐릭터 이동 등) 폴링으로 인풋을 받아야 한다.
  • 만약 연속된 glfwPollEvents() 사이에 인풋이 매우 짧은 시간 동안만 들어온다면 인풋이 씹힐 수도 있는 모양이다. 이 경우 콜백으로 인풋을 받는 게 좋겠다.
  • 폴링으로 처리해야 하는 인풋 종류가 많아지면 CPU에 부하를 줄 수 있으므로 필요한 것들을 제외하고 콜백으로 받는 것이 좋다.
  • 폴링 방식이 짜기도 쉽고 프로그래밍 시에 버그도 덜 일어난다.
  • 한 번의 루프가 도는 시간보다 오랫동안 누르는 키보드 입력(예: 텍스트 입력)같은 경우 콜백으로 받아야 최초 인풋 한 번만 처리된다. 그런데 예를 들어 캐릭터 점프같은 경우는 캐릭터의 상태를 바꾸기 때문에 폴링으로 받아도 최초 인풋 한 번만 처리된다.

FPS 설정하기

  • 업데이트(물리, AI, 비교적 계산 적음) → 고정시간으로 구현
  • 렌더(렌더 하나에 프레임 하나, 셰이더 계산, 비교적 계산 많음) → 유동시간으로 구현, 맥시멈 FPS만 설정

float sec_per_update = 1.0f;
float sec_per_render = 1.0f; // Limiting FPS
float current = 0.0f, previous = 0.0f
float lag = 0.0f, rendered = 0.0f;
int count_update = 0, count_render = 0;
while(!glfwWindowShouldClose(window)) {
	current = glfwGetTime();
	lag += current - previous;
	previous = current;
	
	while(lag >= sec_per_update) {
		lag -= sec_per_update;
		count_update++;
		glfwPollEvents();
		update();
	}

	if(current - rendered >= sec_per_render) {
		rendered = current;
		count_render++;
		render();
		glfwSwapBuffers(window);
	}
}
cout << "Update Per Second: " << count_update / current << endl;
cout << "Render Per Second: " << count_render / current << endl;

만약 sec_per_render를 기기 자체의 주사율보다 빠른 헤르츠를 가지게 잡으면 자동으로 기기의 주사율에 맞추어진다. render() 앞 뒤에 시간을 재서 render에 걸리는 시간을 재보면, 기기가 설정해놓은 주사율에 따라 아직 렌더링하기에는 너무 빠른 상황에서 render()를 맞닥뜨릴 경우 opengl이 알아서 렌더링을 생략해버린다는 것을 알 수 있다. 예를 들어 주사율 60Hz인 내 맥북으로 sec_per_render = 0.01f, 즉 100Hz로 맞추면 렌더함수 호출할 때마다 긴 시간(실제로 렌더링하는 시간)과 짧은 시간(그냥 스킵하는 시간)이 번갈아가면서 뜨는데 이걸로 대략적인 렌더 레이트를 60에 수렴시키는 것을 실험으로 확인할 수 있었다.

update 와 render 의 예시(immediate mode):

float x = 0, y = 0;
void update() {
  float velocity = 0.01f;
  if(glfwGetKey(window, GLFW_KEY_W) == GLFW_PRESS)
		y += velocity;
	if(glfwGetKey(window, GLFW_KEY_A) == GLFW_PRESS)
		x -= velocity;
	if(glfwGetKey(window, GLFW_KEY_S) == GLFW_PRESS)
		y -= velocity;
	if(glfwGetKey(window, GLFW_KEY_D) == GLFW_PRESS)
		x += velocity;
}
void render() {
    glClear(GL_COLOR_BUFFER_BIT);
    glBegin(GL_TRIANGLES);
    glColor3f(1.0f, 0.0f, 0.0f);
    glVertex2f(-0.5f + x, -0.5f + y);
    glVertex2f(0.5f + x, -0.5f + y);
    glVertex2f(0.0f + x, 0.5f + y);
    glEnd();
}

2. GLM의 행렬들 파헤치기

왜 4차원 벡터를 사용하는가?

  • 수학적으로 (일반적으로) M은 affine transformation, V는 rigid motion, P는 (perspective인 경우) projective transformation이다. 이 경우 w의 값은 M,V에서는 1로 유지되지만 P에 의해서는 바뀔 수 있다.
  • 3차원 벡터를 사용하게 되면 3차원 행렬로 처리할 수 있는 변환 중 translation과 perspective projection 를 잃기 때문에 이 두 가지는 행렬을 사용하지 않는 방식으로 따로 구현해줘야 한다. (그렇다고 막 어렵지도 않고 느려지지도 않지만 별로일 듯)
  • 하지만 4차원 벡터를 사용하면 M,V,P에서 사용하는 변환을 한 개념으로 묶어 선형대수의 언어로 통일시켜 처리할 수 있게 된다.
  • 대신 Clip 에서 NDC로 넘어갈 때 perspective division 과정(w를 나누는 것, 어차피 사영공간 정의상 같은 점을 표현하니까 상관x)을 해주게 된다.(이 부분은 opengl의 vertex shader가 자동으로 해주므로 우리의 할 일은 Local을 Clip까지만 무사히 보내는 것)
  • perspective division은 glm에서 (4차든 3차든)행렬을 사용하지 않고 구현되므로 4차원 벡터의 필요성과 상관 없다!
  • 어차피 맨 처음 로드한 모델들의 정점데이터는 3차원으로 저장되고 셰이딩할 때만 4차원 쓰는 거라 메모리가 많이 소비될 일도 없을 것 같은데… 아마…
  • 사원수도 그닥 4차원 벡터를 사용해야 할 이유와는 상관없다.

GLM의 중요한 행렬관련 함수들은 다 어디 들어가 있나?

#include <glm/glm.h>
#include <glm/gtc/matrix_transform.hpp>

를 써주면

#include "../mat4x4.hpp"
#include "../vec2.hpp"
#include "../vec3.hpp"
#include "../vec4.hpp"
#include "../ext/matrix_projection.hpp"
#include "../ext/matrix_clip_space.hpp"
#include "../ext/matrix_transform.hpp"

가 include되므로 matrix_transform, matrix_clip_space, matrix_projection의 함수들을 다 쓸 수 있게 된다.

matrix_projection.hpp

이름이 좀 헷갈리는데 projection matrix 만드는 곳 아니다. model matrix, view matrix, projection matrix을 합치고 perspective division까지 해서 최종 결과물인 NDC를 내놓는 함수 같다. 여기 있는 함수들을 사실 우리가 직접 쓸 일은 없어보인다.

template<typename T, typename U, qualifier Q>
	GLM_FUNC_QUALIFIER vec<3, T, Q> projectZO(vec<3, T, Q> const& obj, mat<4, 4, T, Q> const& model, mat<4, 4, T, Q> const& proj, vec<4, U, Q> const& viewport)
	{
		vec<4, T, Q> tmp = vec<4, T, Q>(obj, static_cast<T>(1));
		tmp = model * tmp;
		tmp = proj * tmp;

		tmp /= tmp.w;
		tmp.x = tmp.x * static_cast<T>(0.5) + static_cast<T>(0.5);
		tmp.y = tmp.y * static_cast<T>(0.5) + static_cast<T>(0.5);

		tmp[0] = tmp[0] * T(viewport[2]) + T(viewport[0]);
		tmp[1] = tmp[1] * T(viewport[3]) + T(viewport[1]);

		return vec<3, T, Q>(tmp);
	}

tmp[0]와 tmp[1] 값 수정하는 부분은 왜 있는 건지 잘 모르겠다…

사원수 관련 함수들은 나중에 다루자.
